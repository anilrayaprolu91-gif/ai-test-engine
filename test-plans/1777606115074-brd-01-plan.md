# Test Plan: BRD-01

## Overview
This test plan covers the requirement for a user to reset their password via email, ensuring that the functionality is working as expected and handles various scenarios.

## Scope
In scope: The test plan includes testing the password reset functionality via email, covering happy path, edge cases, and negative cases.
Out of scope: Testing of other authentication features, such as login and registration, is not included in this plan.

## Test Scenarios

### Scenario 1: Successful Password Reset
**Type:** Happy Path
**Preconditions:**
* The user has a valid account on the application
* The user has access to the email associated with their account
* The user is able to navigate to the application URL: https://practicesoftwaretesting.com/
**Steps:**
1. Given the user is on the login page, they click on the "Forgot Password" link.
2. When the user enters their valid email address and submits the password reset request, they receive an email with a password reset link.
3. Then the user clicks on the password reset link and is redirected to a page where they can enter a new password.
**Expected Result:** The user is able to successfully reset their password and log in to their account with the new password.

### Scenario 2: Invalid Email Address
**Type:** Negative
**Preconditions:**
* The user is on the login page
* The user does not have an account with the entered email address
**Steps:**
1. Given the user is on the login page, they click on the "Forgot Password" link.
2. When the user enters an invalid email address and submits the password reset request, they receive an error message.
3. Then the user checks their email inbox for a password reset email.
**Expected Result:** The user does not receive a password reset email and is displayed an error message indicating that the email address is not associated with an account.

### Scenario 3: Email Address Not Associated with Account
**Type:** Edge Case
**Preconditions:**
* The user has a valid account on the application but with a different email address
* The user is able to navigate to the application URL: https://practicesoftwaretesting.com/
**Steps:**
1. Given the user is on the login page, they click on the "Forgot Password" link.
2. When the user enters an email address that is not associated with their account and submits the password reset request, they receive an error message.
3. Then the user checks their email inbox for a password reset email.
**Expected Result:** The user does not receive a password reset email and is displayed an error message indicating that the email address is not associated with their account.

### Scenario 4: Multiple Password Reset Requests
**Type:** Edge Case
**Preconditions:**
* The user has a valid account on the application
* The user has access to the email associated with their account
* The user is able to navigate to the application URL: https://practicesoftwaretesting.com/
**Steps:**
1. Given the user is on the login page, they click on the "Forgot Password" link and submit a password reset request.
2. When the user submits multiple password reset requests in a short period, they receive multiple password reset emails.
3. Then the user checks their email inbox for the password reset emails and attempts to reset their password using each link.
**Expected Result:** The user is able to successfully reset their password using the most recent password reset link, and the previous links are invalidated.

### Scenario 5: Password Reset Link Expiration
**Type:** Edge Case
**Preconditions:**
* The user has a valid account on the application
* The user has access to the email associated with their account
* The user is able to navigate to the application URL: https://practicesoftwaretesting.com/
**Steps:**
1. Given the user is on the login page, they click on the "Forgot Password" link and submit a password reset request.
2. When the user waits for an extended period before clicking on the password reset link, they are redirected to an error page.
3. Then the user attempts to reset their password using the expired link.
**Expected Result:** The user is unable to reset their password using the expired link and is prompted to request a new password reset.

### Scenario 6: Password Reset with Weak Password
**Type:** Negative
**Preconditions:**
* The user has a valid account on the application
* The user has access to the email associated with their account
* The user is able to navigate to the application URL: https://practicesoftwaretesting.com/
**Steps:**
1. Given the user is on the password reset page, they enter a weak password (e.g., less than 8 characters, no uppercase letter, etc.).
2. When the user submits the weak password, they receive an error message.
3. Then the user checks if they are able to log in with the weak password.
**Expected Result:** The user is not able to reset their password with a weak password and is prompted to enter a stronger password that meets the application's password policy.
